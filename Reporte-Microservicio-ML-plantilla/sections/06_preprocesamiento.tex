\subsection{Esquema de entrada}

El endpoint está diseñado para recibir solicitudes de tipo \texttt{multipart/form-data}, lo que permite gestionar el envío de archivos (PDF, Word o Markdown) de forma eficiente junto con el \texttt{user\_id} del remitente, facilitando así la trazabilidad en la base de datos de auditoría.

\subsection{Validación de entrada estricta}

En el microservicio, decidimos implementar una validación fuerte desde el inicio. El endpoint está configurado para recibir solicitudes de tipo \texttt{multipart/form-data}, esperando un \texttt{user\_id} y un archivo PDF o Markdown. 

Tan pronto como se recibe el archivo, el backend extrae el texto y ejecuta una verificación obligatoria. Buscamos específicamente que el documento contenga secciones fundamentales como "Objetivo General", "Justificación" y "Alcance". Si el texto carece de alguna de estas secciones, la API rechaza inmediatamente la petición con un error \texttt{400 Bad Request}, informando al usuario cuál fue la sección omitida. Esto garantiza que el modelo no procese documentos incompletos ni consuma recursos innecesariamente.

\subsection{Limpieza y anonimización}

Una vez que el documento supera la validación inicial, procedemos a la limpieza del texto. El objetivo en esta etapa es estandarizar la información y, lo más importante, anonimizarla para evitar fugas de privacidad. Para ello, implementamos el siguiente flujo:

\begin{enumerate}
  \item \textbf{Normalización:} Convertimos todo el texto a minúsculas y eliminamos caracteres especiales, saltos de línea innecesarios y puntuación sobrante.
  \item \textbf{Procesamiento NER con SpaCy:} Cargamos el modelo \texttt{es\_core\_news\_sm} de SpaCy para identificar entidades nombradas (Named Entity Recognition).
  \item \textbf{Anonimización:} El sistema recorre el texto y, cada vez que SpaCy detecta una entidad de tipo persona (\texttt{PER}) o ubicación (\texttt{LOC}), la sustituye por un token genérico como \texttt{[PERSONA]} o \texttt{[UBICACION]}.
  \item \textbf{Stopwords:} Finalmente, filtramos las palabras vacías (\textit{stopwords}) comunes en español para reducir el ruido en los datos.
\end{enumerate}

\subsection{Vectorización (Embeddings)}

El paso final del preprocesamiento consiste en transformar el texto limpio en un formato matemático que el algoritmo de clustering pueda interpretar. Para esto, dividimos el texto en fragmentos (chunks) manejables y los procesamos utilizando \texttt{sentence-transformers} con el modelo \texttt{all-MiniLM-L6-v2}. 

Este modelo nos permite generar un vector numérico de 384 dimensiones que captura el significado semántico del texto. De esta manera, el documento completo queda representado en el espacio vectorial y está listo para ser proyectado por UMAP y analizado por HDBSCAN o almacenado en la base de datos ChromaDB.

\subsection{Consistencia entrenamiento vs.\ producción}

Para asegurarnos de que el modelo en producción se trague los datos exactamente igual que como lo entrenamos en el notebook, usamos las mismas dependencias fijas en el \texttt{requirements.txt} del contenedor Docker. Además, el script de limpieza (\texttt{nlp\_service.py}) es el mismo código que corrimos para preprocesar el corpus de 50 proyectos, así que no hay forma de que haya discrepancias.
